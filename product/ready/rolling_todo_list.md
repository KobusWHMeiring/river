# PRD: Global Rolling To-Do List (Refined v2)

## 1. Problem Statement
Managers need a workspace to manage high-priority tasks that aren't strictly tied to a specific date. These "Rolling To-Dos" are often administrative or managerial and should exist in a focused environment separate from the river's daily field operations.

## 2. Strategic Goal
Implement a dedicated Kanban-style To-Do board for managers to track non-dated tasks through "To Do", "Doing", and "Done" stages, with the ability to prioritize via manual sorting.

## 3. Proposed Scope
- **Model Extension (Option A):** Use the existing `Task` model with new fields: `is_rolling`, `todo_status`, `todo_position`, and `is_urgent`.
- **Kanban View (`/todo/`):** A three-column board (To Do, Doing, Done) accessible at `/todo/`.
- **Drag-and-Drop:** Ability to move cards between columns and reorder within a column to manage priority.
- **Urgent Toggle:** A simple "Urgent" checkbox to visually flag critical items.
- **Exclusions:** These tasks will NOT appear on the Global Dashboard, Daily Agenda, or Planners to maintain focus.

## 4. UX/UI Concept
- **Task Form:** A "Rolling Task" toggle. When checked, the "Date" field becomes optional, and the "Urgent" checkbox appears.
- **The Board:** Three columns. Cards show the task name, section (if applicable), and an "Urgent" badge.
- **Interaction:** Dragging a card triggers an AJAX update to its status and position. Sorting within columns is persistent.

## 5. Technical Challenges
- **Sorting Logic:** Integer-based `todo_position` re-indexing.
- **Interactivity:** Integration of `Sortable.js` for smooth drag-and-drop.
- **Task Type Conflict:** Ensuring "Rolling" tasks don't break existing views that expect a `date`.

# Technical Blueprint: Rolling To-Do Kanban

## 1. Database / Models
- **Task Model (`core/models.py`):**
    - `is_rolling = models.BooleanField(default=False, db_index=True)`
    - `is_urgent = models.BooleanField(default=False)`
    - `todo_status = models.CharField(max_length=10, choices=[('todo', 'To Do'), ('doing', 'Doing'), ('done', 'Done')], default='todo', db_index=True)`
    - `todo_position = models.PositiveIntegerField(default=0)`
    - *Architectural Check:* `db_index` is applied to `is_rolling` and `todo_status` for fast board rendering.

## 2. Queries & ORM Strategy
- **Board Fetch:** Use `Task.objects.filter(is_rolling=True).order_by('todo_position')`. Use `select_related('section', 'template')` to avoid N+1 queries for card metadata.
- **Atomic Updates:** Use `transaction.atomic()` when re-indexing positions during a drag-and-drop event to prevent data corruption.

## 3. Services / Business Logic
- **Task Services (`core/services/task_services.py`):**
    - `def move_todo_task(task_id: int, new_status: str, new_index: int) -> None:`
        - Logic to update the status and shift positions of all other tasks in the target column.

## 4. Views / API Endpoints
- **TodoKanbanView (`core/views.py`):** Standard `ListView` filtering for `is_rolling=True`.
- **TodoUpdateAPI (`core/views.py`):** A POST endpoint for AJAX calls from `Sortable.js`.
- **TaskForm (`core/forms.py`):**
    - Make `date` field `required=False`.
    - Add `clean()` logic: if `is_rolling` is False, `date` MUST be provided.

## 5. Step-by-Step Implementation Checklist
1. **Model Update:** Add fields to `Task` and generate migrations.
2. **Service Layer:** Implement `move_todo_task` logic.
3. **Template:** Create `core/todo_kanban.html` with three column containers.
4. **JS Integration:** Add `Sortable.js` via CDN and write the `onEnd` handler to POST to the API.
5. **View & URL:** Register `TodoKanbanView` and the update API.
6. Form Update: Adjust TaskForm validation and template visibility.

- Target File: `core/views.py` (Update `TaskCreateView` and `TaskUpdateView`)

- [ ] `Task` model includes `is_rolling`, `is_urgent`, `todo_status`, and `todo_position`.
- [ ] `TaskForm` displays an "is_rolling" toggle.
